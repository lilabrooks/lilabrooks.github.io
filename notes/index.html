Questions · decisions · checks

Working methods, written down.

These are the questions I use to keep a small software project understandable. They change when the repositories give me a better answer.

Before the code

Questions worth answering early.

A little written reasoning at the start makes later changes easier to judge. I keep the answers close to the source and revise them when the evidence changes.

  1. 01 · Problem

    What recurring problem deserves a tool?Problem framing

    I try to name the repeated decision or task, who encounters it, and what a useful answer would look like. A narrow problem gives the first version a boundary.

  2. 02 · Boundary

    Where should responsibility stop?System shape

    Inputs, outputs, state, upstream services, and failure behavior belong in the first sketch. The boundary should be small enough to explain without hiding the difficult parts.

  3. 03 · Evidence

    What would prove the behavior works?Verification

    I prefer tests at the application boundary, recorded upstream responses, and deterministic demos. The check should still mean something when the network or calendar changes.

  4. 04 · Change

    Which decisions need to survive the session?Project memory

    Specifications record behavior. Architecture records explain choices and consequences. Source mappings help the next change find the small set of context it needs.

  5. 05 · Operation

    How will a failure explain itself?Operating behavior

    Startup checks, useful error messages, stored results, and visible decision reasons make a small service easier to trust and repair.

During the work

Keep the reasoning near the change.

The method is intentionally small. Each piece should help with a decision, a review, or a later repair.

Write the contract

Record inputs, outputs, failure cases, and acceptance checks before implementation spreads the behavior across files.

Record consequential choices

Keep the options, decision, and consequences together. A short record is enough when it answers why the code took this shape.

Test without outside luck

Replace live services at the boundary. Fixed fixtures and clocks keep tests useful when upstream data changes.

Leave an honest edge

Write down what the current design leaves out, which assumptions are temporary, and what would force the boundary to move.

In context

The project notes contain the details.

Each note connects these questions to one repository, including the choices that remain open.
