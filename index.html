Claude Code · Open Knowledge Format

A repository that carries its own intent.

The Claude OKF repo kit gives Claude Code a stable account of the goal, the behavior it must preserve, the decisions already made, and the work still open.

Working premise

The conversation is temporary. The repository is the memory.

The intent

Code needs the reasoning around it.

A source tree shows what exists today. It rarely explains why a boundary exists, which compromise was accepted, whether a change contradicts the goal, or where the last session stopped.

The kit treats project intent as maintained repository data. Its document conventions build on the Open Knowledge Format; the goal loop, source map, and Claude Code guardrails make those conventions operational.

Claude Code begins each session with a compact view of that knowledge. It reads the full documents connected to the current task, makes the smallest change that satisfies the goal, and leaves the repository with a current account of what changed and why.

The knowledge model

Different questions belong in different records.

The structure is deliberately small. Each part answers one kind of question, and the source map connects the pieces needed for a change.

The goal sets direction

The goal names the problem, target state, success criteria, non-goals, constraints, and ordered milestones. Its scope and order remain the owner’s decisions.

Specs state behavior

Specifications describe the contracts the implementation must preserve: inputs, outputs, failure cases, and acceptance conditions.

ADRs preserve reasoning

Architecture decision records hold the context, chosen option, consequences, and revisit trigger for decisions that should outlive one session.

The map narrows context

A source-to-knowledge map points each part of the codebase to its governing specs and decisions. This keeps each session’s reading set small and specific.

The mechanism

Context becomes a repeatable working loop.

The kit turns the repository’s knowledge into session behavior. It provides a path from stated intent to a verified change and a record the next session can trust.

01Load the goal and indexes
02Read the knowledge mapped to the task
03Change code and governing records together
04Verify, log, and continue
  1. Session context

    The project instructions preload the goal plus the spec and ADR indexes. Full documents stay on disk until the current work calls for them.

  2. Goal-directed work

    A general request to continue means taking the first unfinished milestone, reading its governing knowledge, and working until its stated verification passes.

  3. Knowledge sync

    Behavior changes carry matching spec, decision, mapping, or log updates. A stop-time check catches code changes that leave the repository’s account behind.

  4. Recorded decisions

    Dependencies, persistence, APIs, security boundaries, and deployment choices become proposed ADRs. The owner can review them without interrupting each implementation step.

  5. Completion and resumption

    Milestones close after verification. A final acceptance pass uses the project like a new user would. Checkboxes, logs, ADRs, commits, and in-flight changes give the next session a precise restart point.

Control boundaries

The owner keeps the consequential decisions.

The kit gives Claude Code room to work inside a written contract. Changes to that contract remain visible and reviewable.

  • Owner

    Goal and scope

    The owner sets the goal, changes milestone order, and decides when a non-goal should move.

  • Agent

    Implementation

    Claude Code makes implementation choices inside the goal, accepted decisions, specifications, and standing guardrails.

  • Review

    Proposed decisions

    Decision-shaped changes are written down with their consequences and rollback conditions, then left for owner review.

  • Consent

    External effects

    Publishing, deployment, destructive changes, and outside actions with side effects wait for explicit approval.

The enforcement boundary matters. Documentation sync and local environment-file read denial are mechanical gates. Test discipline, security rules, and consent rules are instructions Claude Code is expected to follow. Repositories that require guaranteed gates should enforce them in CI and branch protection.

Evidence from use

The kit changes when real repositories find a weak spot.

Three dogfood repositories exercised the model with a packaged Python CLI, a small web application, and an always-on scheduled service. Their findings became changes to the kit’s interviews, hooks, upgrade policy, mapping guidance, and review loop.
