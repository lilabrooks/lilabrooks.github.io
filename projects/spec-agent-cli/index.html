Spec Agent CLI

A Python CLI that combines a Markdown specification, reusable agent skills, and a chosen model provider to plan and write a working command-line project.

The problem

A free-form prompt can produce code, though it rarely captures the whole contract of a command-line tool. Inputs, outputs, failure behavior, packaging, and acceptance tests need a stable shape that survives beyond one model call.

Spec Agent CLI makes the specification a first-class input. The user writes the command surface and expected behavior in Markdown, attaches working skills when needed, previews the generated file plan, then chooses whether those files should be written.

The system

01Spec and skills
02Provider-neutral request
03FILE block parser
04Preview or apply

The CLI resolves a spec from a slug or path, validates its required sections, loads selected skills, and assembles the agent request. A provider adapter returns text. The build command parses explicit FILE: path blocks, shows the planned output, and writes files only when the user supplies --apply.

The installed package bundles its default specs and skills, so the same workflow works outside the source checkout.

Safety boundaries

  1. 01

    Validate before the model call. A strict mode stops on an incomplete spec before credentials or model time are used.

  2. 02

    Preview before writing. The default build command prints the file plan and leaves the filesystem alone.

  3. 03

    Protect existing work. Apply refuses to overwrite a file unless the user also supplies the force flag.

  4. 04

    Keep provider choice at the edge. The core agent logic receives a small language-model interface instead of constructing vendor clients directly.

Why the provider boundary matters

A small LanguageModel protocol defines the completion contract. Provider adapters and a registry handle vendor-specific clients. The CLI, spec loader, skill loader, parser, and tests depend on the protocol.

The default echo adapter needs no credentials and makes the command surface testable without network calls. Real provider SDKs are optional extras selected by configuration.

Tradeoffs

The file parser depends on the model following an explicit response contract. A response without valid file blocks can still be shown to the user, though there is nothing safe to apply.

The generator does not decide what a project should look like. The specification and attached skills determine whether the result is one script or an installable package. That flexibility puts more responsibility on the author of the spec.

How the work is verified

The repository uses pytest for behavior, Ruff for code quality, and mypy in strict mode. CI covers supported Python versions and checks the OKF documentation bundle alongside the source.
