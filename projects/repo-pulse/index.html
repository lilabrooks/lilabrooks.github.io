Repo Pulse

A local dashboard that turns six GitHub tabs into one readable answer about a public repository’s recent activity.

The problem

GitHub has the data needed to judge whether a public repository is active, quiet, dormant, or archived. The answer is scattered across the main page, issues, pull requests, releases, and commit history.

Repo Pulse accepts an owner and repository name, gathers those signals, and returns one summary: stars, issue and pull request counts, latest release, 30-day commit activity, and a pulse verdict with clear rules.

The system

The hand-written dashboard sends one request to the FastAPI service. The service owns GitHub communication, normalizes the responses, separates open issues from open pull requests, calculates the activity window, and returns one stable response contract.

A process-local cache holds each repository summary for five minutes. Repeated views stay fast and avoid spending API calls on unchanged data.

Decisions that shape it

  1. 01

    Keep the browser thin. GitHub credentials and response normalization stay behind the API boundary.

  2. 02

    Use a small cache. Repository data tolerates a five-minute delay, and the cache removes repeated upstream work without adding persistent infrastructure.

  3. 03

    Make the verdict mechanical. Archived overrides every other state. Recent pushes produce active, older pushes produce quiet, and long inactivity produces dormant.

  4. 04

    Accept the formats people paste. The input parser handles owner/repo, GitHub URLs, trailing .git, and extra URL parts before the API sees the identifier.

How the work is verified

Every GitHub interaction is replaced by an httpx.MockTransport in the test suite. Tests run offline and cover the application boundary without depending on rate limits, repository changes, or credentials.

The repository keeps its goal, API contract, frontend behavior, and architecture decisions under docs/. CI checks the tests and the documentation mapping on every change.

Tradeoffs

The cache disappears when the process restarts and cannot coordinate across multiple workers. That is a deliberate boundary for a local tool. A hosted version would need shared caching, credential handling, and a clearer operating model.

The current design accepts GitHub repository identifiers. Resolving arbitrary project homepages would add discovery rules and ambiguous failure cases, so that idea remains recorded as a proposed scope change.
