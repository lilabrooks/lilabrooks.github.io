Skywatch

A local service that joins ISS pass predictions with cloud forecasts, explains the verdict, and sends mail only when tonight is worth looking up.

The problem

An ISS pass prediction answers when the station crosses the sky. It does not answer the practical question: will the station be visible, high enough to see clearly, and under a usable sky at this location?

Skywatch combines pass data and hourly cloud cover into an explainable go, maybe, or skip decision. It keeps the negative answers visible on a local status page and stays quiet unless the next 24 hours contain a worthwhile pass.

The system

01Fetch predictions
02Fetch cloud cover
03Join and judge
04Store, show, notify

A scheduler fetches ISS pass predictions from sat.terrestre.ar and the hourly forecast from Open-Meteo. The service stores the results in SQLite, matches each pass window to its overlapping forecast hours, calculates a verdict, updates a status page, and decides whether a digest should be sent.

Stored results keep the page useful during an upstream outage. Configuration comes from environment variables and an optional local .env file. Invalid configuration fails at startup with one collected error report.

The decision rules

  1. 01

    Skip when the station is not visible, the pass peaks below 25°, or the worst overlapping cloud forecast is above 70%.

  2. 02

    Go when the pass clears the visibility and elevation gates and cloud cover is at or below 30%.

  3. 03

    Maybe when cloud cover falls between those thresholds or the forecast does not cover the pass window yet.

  4. 04

    Notify once. The service sends at most one digest per day and only when a go or maybe pass exists in the next 24 hours.

The status page includes the reason behind each verdict. A useful decision service should explain silence as well as action.

How the work is verified

The application runtime uses the Python standard library. Tests use recorded upstream fixtures and run offline. The demo shifts fixture timestamps so a clear-sky pass always lands tonight, then exercises the real decision and SMTP path against a local inbox.

This makes the demo deterministic. Weather and orbital timing cannot turn a known test into a shrug.

Tradeoffs

Latitude and longitude are the v1 location boundary. Address lookup would add a geocoding provider, more failure modes, and another privacy choice.

The verdict uses fixed thresholds. They are easy to inspect and tune, though they cannot express every observer’s preference or local viewing condition. The reasons remain explicit so the user can disagree with the decision.
