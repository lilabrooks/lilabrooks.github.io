Claude OKF repo kit

The problem

A coding agent can read source code and still miss the reason the system looks the way it does. Goals, constraints, rejected options, and verification habits often live in chat history or in someone’s head.

The repo kit moves that knowledge into version control. Claude Code starts with the project goal and knowledge indexes in context, then follows the relevant specifications and architecture decisions when it changes governed files.

The knowledge model

01Goal and milestones
02Behavior specs
03Architecture decisions
04Source mappings

CLAUDE.md defines the project rules and imports the goal plus the spec and ADR indexes. docs/GOAL.md holds the problem, target state, success criteria, and milestone backlog. Specs describe behavior. ADRs record decisions and consequences. An OKF map connects source paths to the knowledge that governs them.

Hooks watch for code changes that leave the related knowledge stale. A small command-line helper checks draft specs, stale references, and places where an architecture decision may be needed.

The working loop

  1. 01

    State the goal and ordered milestones before asking an agent to implement features.

  2. 02

    Write behavior contracts and architecture decisions beside the source they govern.

  3. 03

    Have the agent read the mapped knowledge before editing a source area.

  4. 04

    Run repository checks that catch code changes without matching knowledge updates.

The goal is a smaller gap between what the team means and what the agent can see.

Decisions that shape it

Version the reasoning

Project intent changes with the code and should travel through the same review history. Markdown keeps the knowledge readable to people, agents, and ordinary repository tooling.

Map knowledge to source

A large documentation folder is easy to ignore. The OKF map narrows the reading set to the specs and decisions connected to the files being changed.

Use guardrails as feedback

The hooks flag stale knowledge at the moment it matters. They turn documentation upkeep into part of the implementation loop.

Tradeoffs

The structure asks for writing before and during implementation. Small throwaway experiments may not earn that cost. The kit is most useful when a repository will change over time, several decisions need to survive, or agent sessions need a shared memory.

Hooks can detect likely gaps. They cannot prove that a specification is correct or that an ADR captures the real reasoning. Review still belongs to people.
